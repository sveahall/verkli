Critical Findings

Critical Secrets committed to repository

.env.local (lines 16-54)

Production/development secrets are committed to version control despite .gitignore listing the file. Exposed keys include: Supabase service role key, Stripe secret key, Resend API key, and ElevenLabs API key.

Impact: Complete compromise of backend (service role key bypasses all RLS), billing system, email sending, and TTS services.

Fix: Rotate all keys immediately. Remove from git history with git filter-repo. Add a pre-commit hook to scan for secrets.

Critical 13 dependency vulnerabilities (10 High, 3 Moderate)

apps/web/package.json

Key vulnerabilities: Next.js HTTP request smuggling, CSRF null-origin bypass on Server Actions, unbounded image disk cache, plus issues in undici, xml2js, tar, and flatted.

Fix: Run npm audit fix and update Next.js to the latest patch release.

Architecture

Excellent Monorepo structure

Clean npm workspaces layout with apps/web, apps/worker, and four shared packages (db, shared, ui, config). Root dependencies are minimal. The explicit Prisma ban (check:no-prisma script) keeps the data layer consistent.

Excellent App Router and route groups

Well-organized route groups separate concerns cleanly: (public-author), (public-reader), (app-author), (app-reader), (auth), (selector), (reader-browse). Auth checks happen at the layout level, preventing unauthorized access early. Over 10 loading.tsx files provide skeleton states for streaming.

Good Data layer (Supabase)

Centralized client creation (createClient() for SSR, createAdminClient() for server-only). A service layer in /lib/books/service.ts provides type-safe operations with ServiceResult. One issue: GlobalNavbar.tsx makes Supabase queries client-side — this should be a server action or API route.

Excellent Worker architecture

Seven BullMQ queues with centralized queue names, exponential backoff retries, duplicate detection, and graceful degradation when Redis is unavailable. Unified startup script starts all workers in one process with proper SIGINT/SIGTERM handling.

Missing dead letter queue

Failed jobs log errors but have no DLQ for visibility. After 2 retries, jobs silently fail with no way to review or replay them.

Excellent API routes

Every endpoint validates input with Zod schemas (strict enums, numeric ranges, string length limits). Auth checks happen before business logic. Centralized error keys (/lib/api-errors.ts, 100+ codes) map to localized messages. Rate limiting on sensitive endpoints (waitlist, comments, billing checkout) using a Redis-backed token bucket algorithm.

Code Quality

Excellent TypeScript strictness

Only 1 instance of any in the entire codebase (book-translation.ts, acceptable use). Strict mode enabled. Strong typing throughout, especially on Supabase queries and API responses.

Security

Excellent Authentication and authorization

Timing-safe comparison for admin token validation. Role checks query the database (not client-writable metadata). Separate helpers for admin, author, and ops access. Beta lock mechanism gates unauthorized access during beta phase. CSRF protection in middleware validates Origin headers on mutating requests.

Excellent Security headers

Comprehensive CSP policy, X-Frame-Options: DENY, X-Content-Type-Options: nosniff, strict referrer policy, and restrictive permissions policy. Stripe webhooks use signature verification and are excluded from CSRF checks.

Good Input validation and XSS prevention

Supabase SDK prevents SQL injection. LIKE wildcards are properly escaped in search. File uploads validated by extension whitelist (epub, docx, html, txt, pdf) and 50MB size limit. DOMPurify sanitizes user HTML in newsletter rendering. No unsanitized user content rendering found.

Missing MIME type validation on uploads

File uploads only check extensions, not magic bytes. A malicious file with a renamed extension could bypass validation.

Some admin endpoints lack rate limiting
/api/admin/users, /api/admin/author-applications

Admin routes don't have per-user rate limiting, making them vulnerable to enumeration attacks.

RLS policies unverified

No SQL migration files found in supabase/migrations/. Row Level Security policies should be verified in the Supabase console to ensure proper access control on all tables.

Performance

Good Bundle and images

Bundle analyzer configured. Server-external packages (epub, pdf-parse, bullmq) prevent client-side bundling. Next.js Image component used in 33 files with responsive sizes props. One raw img tag found in CampaignWizard.tsx line 260.

Excellent Data fetching patterns

Batch profile fetching via getProfilesByUserIds() with deduplication prevents N+1 queries. Parallel Promise.all() used for independent queries. Selective column queries (.select("id, title, status")) throughout — no SELECT * observed.

Needs work React optimization

Only 30 out of 91 component files use useMemo/useCallback. Frequently-rendered list components like BookCard aren't memoized. Inline event handlers in render paths create new functions each render. Only 10 Suspense boundaries across 1,173 files — granular streaming is underutilized.

69 of 91 components marked "use client"

High ratio of client components increases JS bundle size. Some could likely be refactored to server components with interactive children extracted to smaller client wrappers.

Polling-based job tracking instead of Supabase Realtime

Workers update the database but the UI polls API endpoints on intervals. Supabase Realtime subscriptions would provide instant feedback with less network overhead.

Shared Packages

Good packages/shared

Well-defined exports for contracts (API response types, DTOs), schemas (Zod validation), and constants (roles, statuses, limits). Prevents magic strings and ensures consistent validation across routes.

packages/ui and packages/db are stubs

Both export empty objects. UI components live in apps/web/src/components/ and types live in apps/web/src/lib/supabase/types.ts. This is fine for a single app but will cause duplication if the monorepo scales.

What's Working Well

This codebase has strong fundamentals. The architecture is clean, TypeScript usage is nearly flawless (1 any in 1,173 files), and the API layer has excellent discipline with Zod validation and centralized error codes on every endpoint. The worker system is well-designed with graceful degradation, and the auth/authorization model is solid with database-backed role checks and timing-safe comparisons. The structured logging makes debugging in production much easier than most codebases. The main improvements needed are around the two critical security items (secrets + deps), splitting a handful of large components, and adding more granular streaming with Suspense.
